## Tech Stack (actual ‚Äî this diverged from the original assumed defaults, keep this in sync)
- Frontend (web): React 18 + TypeScript + Vite. **Mid-migration to Tailwind CSS** (added for the Agni Operational System redesign, see UI Redesign section below) ‚Äî `tailwind.config.js` has `corePlugins.preflight: false` on purpose, and the legacy hand-written `src/index.css` (~1036 lines) still styles every not-yet-migrated page. Don't "clean up" the legacy CSS or flip preflight on until Stage 2 (below) is done.
- Frontend (mobile): Capacitor wraps the same React app as a native Android shell (`frontend/android/`) ‚Äî not React Native/Flutter. Used today for the driver app's background GPS tracking. iOS not set up.
- Backend: almost none. One Cloud Function (`functions/`, `chatAssistant`, **deployed and live**) proxies Gemini API calls for the AI Chat Assistants ‚Äî the only place a paid API key needs to stay server-side. Everything else is still client-side + `firestore.rules`. The project is now on Firebase's **Blaze** (pay-as-you-go) plan ‚Äî Spark can't run Cloud Functions at all, so don't assume Spark-only constraints apply anymore for backend work, though the client itself should still default to client-side Firestore logic (see Conventions).
- Database: Cloud Firestore (also the realtime layer, via `onSnapshot` ‚Äî no sockets)
- Auth: Firebase Authentication (Role-Based Access Control per section 9)
- Maps/GPS: **Leaflet + OpenStreetMap** (not Google Maps ‚Äî no API key needed/wanted). Route drawing via OSRM + Nominatim, called once at admin bus-save time.
- Hosting: Firebase Hosting

## Conventions
- Keep each Claude Code session scoped to ONE phase/feature ‚Äî do not attempt multiple phases in a single session.
- Reference `PROJECT_SPEC.md` section numbers when asking Claude Code to build a feature (e.g. "build section 2's Missed Bus Recovery flow").
- Update the "Build Phases" checklist and "Current phase" line above as work completes ‚Äî they drifted out of sync with reality once before, keep them honest.
- Firestore security rules must reflect the 6 roles defined in section 1 at all times.
- There is no location-history collection, so anything needing historical GPS data (speed, harsh braking, a real ETA-vs-actual "delay") needs a deliberate architecture decision first ‚Äî flag it rather than assuming it can just be added.
- The `chatAssistant` Cloud Function is the one exception to "everything is client-side" ‚Äî it exists solely because a paid LLM API key can't be shipped to the browser. Don't add a second one without the same reasoning; prefer client-side Firestore logic by default.
- **Never ask the user to paste an API key or other secret into chat.** Use `firebase functions:secrets:set <NAME>` (they run it, type the value at the prompt, it never appears in the conversation) or point them at `functions/.secret.local` (gitignored) for emulator-only testing.
- To verify a driver- or admin-only flow without real credentials: `firebase emulators:start --only firestore,auth` (add `,functions` if testing the chat assistants ‚Äî put a placeholder value in `functions/.secret.local` first, e.g. `GEMINI_API_KEY=placeholder`, since the function needs *a* string to be defined even though a fake one will only prove the request reaches Gemini, not that it gets a real reply), then `VITE_USE_EMULATOR=true npm run dev` (see `frontend/src/lib/firebase.ts`) and seed test accounts with `firebase-admin` pointed at `FIRESTORE_EMULATOR_HOST`/`FIREBASE_AUTH_EMULATOR_HOST` ‚Äî bypasses the self-signup role restrictions entirely, with zero risk to the live project.
- `createStaffAccount` in `frontend/src/services/firestore.ts` (backs driver/maintenance/parent/principal account creation) uses a **secondary** Firebase app instance so the admin doesn't get signed out. That secondary app does NOT automatically inherit `../lib/firebase.ts`'s emulator wiring ‚Äî it needs its own `connectAuthEmulator` call gated on the same `VITE_USE_EMULATOR` check, or account creation through the admin UI silently hits **production** Auth even during emulator testing (this actually happened once ‚Äî caught via the `EMAIL_EXISTS` response pointing at `identitytoolkit.googleapis.com` instead of `127.0.0.1:9099`, fixed, and the stray production account was deleted). If you ever add another secondary-app pattern, remember to wire the emulator connection on it too.
