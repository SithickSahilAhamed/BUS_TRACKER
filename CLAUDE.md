# AGNI Smart Fleet AI — Project Memory

Full specification: see `PROJECT_SPEC.md` (repo root — contains all 9 sections: user roles, student module, driver module, admin dashboard, fleet maintenance, AI analytics, notifications, reports, security). Future Enhancements were intentionally excluded from this build — do not implement them unless explicitly asked.

Deployed as **ACT To Go** for Agni College of Technology — see `README.md` for setup/deploy instructions.

## Tech Stack (actual — this diverged from the original assumed defaults, keep this in sync)
- Frontend (web): React 18 + TypeScript + Vite (not Tailwind)
- Frontend (mobile): Capacitor wraps the same React app as a native Android shell (`frontend/android/`) — not React Native/Flutter. Used today for the driver app's background GPS tracking. iOS not set up.
- Backend: almost none. One Cloud Function (`functions/`, `chatAssistant`, **deployed and live**) proxies Gemini API calls for the AI Chat Assistants — the only place a paid API key needs to stay server-side. Everything else is still client-side + `firestore.rules`. The project is now on Firebase's **Blaze** (pay-as-you-go) plan — Spark can't run Cloud Functions at all, so don't assume Spark-only constraints apply anymore for backend work, though the client itself should still default to client-side Firestore logic (see Conventions).
- Database: Cloud Firestore (also the realtime layer, via `onSnapshot` — no sockets)
- Auth: Firebase Authentication (Role-Based Access Control per section 9)
- Maps/GPS: **Leaflet + OpenStreetMap** (not Google Maps — no API key needed/wanted). Route drawing via OSRM + Nominatim, called once at admin bus-save time.
- Hosting: Firebase Hosting

- [x] **Phase 2** — Driver module: start/end trip, GPS sharing (web + native Android background tracking), next-stop reminder, waiting-student count, boarding roster, incident/damage reporting all done. - [x] **Phase 3** — Admin Dashboard: bus CRUD + route drawing (now with seating capacity), driver CRUD, live tracking map, student→bus/stop assignment, driver reports, live Attendance (assigned/boarded/likely-absent, computed the same route-projection way as Next Stop — not a historical log), Missed Bus Recovery (student requests + admin approve/deny), Breakdown Management (suggest buses with free seats, bulk-reallocate riders), campus entry/exit geofence log all done. Dedicated Stop/Route entities as their own collections deliberately **not done** — the current bus-owns-its-route model works fine at this fleet's scale and decoupling them is a large rewrite the spec's CRUD wording doesn't actually require.
- [x] **Phase 4** — Fleet Maintenance: `maintenance` role now has real sign-up (admin-created, like drivers) + its own panel. Vehicle Profile (model/manufacturer/registration/engine/chassis + insurance/permit/FC/PUC expiry) and a fleet-wide "needs attention" expiry summary, Fuel Management with derived mileage, Maintenance History (also covers Tyre/Battery Management — one collection with a category, not three), Service Schedule (admin or maintenance sets a due date, same due-soon summary), and Repair Requests done. Repair Requests deliberately **reuses** the driver's Phase 2 damage reports rather than a separate collection — "driver submits, maintenance closes" is the same lifecycle already built, just with a `repairNotes` field and maintenance-scoped rules added.
- [x] **Phase 5** — AI Analytics: Fuel/Route/Student/Driver Analysis done as plain client-side aggregation (Admin → Analytics), including a new `trips` collection logged once at End Trip (duration only — no GPS trail, no speed/harsh-braking, see below). "Predictive Maintenance" reuses Phase 4's date-based due-soon reminders rather than a fabricated ML prediction — there's no sensor data or history to base one on. AI Chat Assistants: **live in production.** Student/Driver/Admin each get a floating `ChatAssistant` widget backed by a real Gemini API call (model `gemini-flash-latest`) through the `chatAssistant` Cloud Function (client sends a role-appropriate data summary as `context`; the function only adds the persona + proxies the key). Principal AI is not built — no Principal dashboard exists yet (Phase 6). Project is on Blaze; `GEMINI_API_KEY` is set via `firebase functions:secrets:set`; confirmed with a real end-to-end call against the live function returning a genuine on-persona Gemini reply, not just the emulator/placeholder-key plumbing check.
- [x] **Phase 6** — Notifications: in-app Notification Center (`notifications` collection, written only by 6 new Cloud Function Firestore triggers — `onSosAlertCreated`, `onReportCreated`, `onReportUpdated`, `onMissedBusRequestCreated`, `onMissedBusRequestUpdated`, `onUserAssignmentChanged`; read via a `NotificationBell` merging uid-targeted + role-targeted live subscriptions) plus an SOS button (student/professor/driver) feeding the same pipe. Reports: PDF/Excel export (`jspdf`+`autotable`, `xlsx`) wired into all 4 Analytics sections via a reusable `ExportButtons`. ## Conventions
- Keep each Claude Code session scoped to ONE phase/feature — do not attempt multiple phases in a single session.
- Reference `PROJECT_SPEC.md` section numbers when asking Claude Code to build a feature (e.g. "build section 2's Missed Bus Recovery flow").
- Update the "Build Phases" checklist and "Current phase" line above as work completes — they drifted out of sync with reality once before, keep them honest.
- Firestore security rules must reflect the 6 roles defined in section 1 at all times.
- There is no location-history collection, so anything needing historical GPS data (speed, harsh braking, a real ETA-vs-actual "delay") needs a deliberate architecture decision first — flag it rather than assuming it can just be added.
- The `chatAssistant` Cloud Function is the one exception to "everything is client-side" — it exists solely because a paid LLM API key can't be shipped to the browser. Don't add a second one without the same reasoning; prefer client-side Firestore logic by default.
- **Never ask the user to paste an API key or other secret into chat.** Use `firebase functions:secrets:set <NAME>` (they run it, type the value at the prompt, it never appears in the conversation) or point them at `functions/.secret.local` (gitignored) for emulator-only testing.
- To verify a driver- or admin-only flow without real credentials: `firebase emulators:start --only firestore,auth` (add `,functions` if testing the chat assistants — put a placeholder value in `functions/.secret.local` first, e.g. `GEMINI_API_KEY=placeholder`, since the function needs *a* string to be defined even though a fake one will only prove the request reaches Gemini, not that it gets a real reply), then `VITE_USE_EMULATOR=true npm run dev` (see `frontend/src/lib/firebase.ts`) and seed test accounts with `firebase-admin` pointed at `FIRESTORE_EMULATOR_HOST`/`FIREBASE_AUTH_EMULATOR_HOST` — bypasses the self-signup role restrictions entirely, with zero risk to the live project.
- `createStaffAccount` in `frontend/src/services/firestore.ts` (backs driver/maintenance/parent/principal account creation) uses a **secondary** Firebase app instance so the admin doesn't get signed out. That secondary app does NOT automatically inherit `../lib/firebase.ts`'s emulator wiring — it needs its own `connectAuthEmulator` call gated on the same `VITE_USE_EMULATOR` check, or account creation through the admin UI silently hits **production** Auth even during emulator testing (this actually happened once — caught via the `EMAIL_EXISTS` response pointing at `identitytoolkit.googleapis.com` instead of `127.0.0.1:9099`, fixed, and the stray production account was deleted). If you ever add another secondary-app pattern, remember to wire the emulator connection on it too.
